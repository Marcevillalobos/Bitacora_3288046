PROYECTO FINAL DE FUNDAMENTOS DE SOFTWARE

1. Análisis de Requisitos

Es la fase inicial del desarrollo de software donde se identifican, documentan y validan las necesidades del cliente o usuario final. Su objetivo es definir qué debe hacer el sistema antes de diseñarlo o implementarlo.


2. Requisitos Funcionales

Son las características específicas que describen lo que el sistema debe hacer, como procesos, cálculos, almacenamiento de datos y flujos de información.


3. Requisitos No Funcionales

Se refieren a las cualidades del sistema, como rendimiento, seguridad, usabilidad, escalabilidad y disponibilidad, que determinan la experiencia del usuario y la eficiencia técnica.


4. Caso de Uso

Es una descripción estructurada de cómo un usuario interactúa con el sistema para lograr un objetivo. Representa escenarios y define el comportamiento esperado del software.


5. Diagrama de Casos de Uso

Herramienta visual de UML que muestra las interacciones entre actores externos y los diferentes casos de uso que ofrece el sistema.


6. Modelado de Procesos

Consiste en representar gráficamente los flujos de trabajo de una organización o sistema, identificando entradas, salidas, actores y procesos internos.


7. Diseño de Software

Es la fase donde se estructura la solución técnica del sistema, definiendo la arquitectura, componentes, interfaces y cómo se interconectan entre sí.


8. Arquitectura de Software

Es la organización fundamental del sistema en términos de sus componentes, relaciones y principios que guían su evolución y desarrollo.


9. Patrón de Diseño

Es una solución reutilizable y probada para resolver problemas recurrentes en el diseño de software, como Singleton, Factory o MVC.


10. Diagrama de Clases

Es una representación UML que muestra la estructura estática del sistema mediante clases, atributos, métodos y relaciones entre ellas.


11. Programación Orientada a Objetos (POO)

Paradigma de programación basado en objetos que combinan datos (atributos) y comportamientos (métodos), promoviendo modularidad y reutilización.


12. Encapsulamiento

Principio de la POO que restringe el acceso directo a los datos de un objeto, protegiéndolos mediante métodos públicos controlados.


13. Herencia

Mecanismo que permite a una clase derivada reutilizar atributos y métodos de otra clase base, promoviendo la reutilización de código.


14. Polimorfismo

Capacidad de un objeto para comportarse de diferentes formas dependiendo del contexto, facilitando la extensibilidad del software.


15. Cohesión

Medida de cómo los elementos de un módulo están relacionados entre sí. Alta cohesión implica que el módulo tiene una única responsabilidad.


16. Acoplamiento

Grado de dependencia entre módulos de software. Un bajo acoplamiento es deseable porque aumenta la flexibilidad y mantenibilidad.


17. Ciclo de Vida del Software

Conjunto de fases que sigue un proyecto de software desde su concepción hasta su retiro, como análisis, diseño, implementación, pruebas y mantenimiento.


18. Modelo en Cascada

Metodología secuencial de desarrollo en la que cada fase debe completarse antes de iniciar la siguiente, con poca flexibilidad para cambios.


19. Metodología Ágil

Enfoque iterativo e incremental que promueve entregas rápidas, adaptabilidad y colaboración constante con el cliente.


20. Scrum

Marco de trabajo ágil que organiza el desarrollo en ciclos llamados sprints, con roles definidos como Product Owner, Scrum Master y Equipo de Desarrollo.


21. Kanban

Es una metodología ágil que utiliza tableros visuales con columnas para gestionar el flujo de trabajo, optimizar tiempos y limitar el trabajo en curso.


22. Iteración

Es un ciclo de trabajo en el desarrollo ágil en el que se planifican, desarrollan, prueban y entregan funcionalidades del sistema de forma incremental.


23. Product Backlog

Lista priorizada de todas las funcionalidades, mejoras y correcciones que requiere un producto, administrada por el Product Owner en Scrum.


24. Sprint

Periodo de tiempo corto y fijo (generalmente de 2 a 4 semanas) en el que el equipo desarrolla un conjunto de funcionalidades entregables.


25. Daily Meeting

Reunión diaria breve en Scrum donde el equipo sincroniza actividades, revisa avances y detecta impedimentos.


26. Prototipo

Representación inicial del software, ya sea en forma de maqueta o versión limitada, para validar ideas y recopilar retroalimentación.


27. Wireframe

Esquema visual simplificado de una interfaz de usuario que muestra la estructura y disposición de los elementos sin entrar en detalles gráficos.


28. Mockup

Es una representación estática más detallada de la interfaz que incluye diseño visual, colores y tipografía para mostrar cómo se verá el sistema.


29. Prueba de Software

Conjunto de actividades que buscan verificar que el sistema cumple con los requisitos y funciona correctamente en diferentes escenarios.


30. Prueba Unitaria

Evaluación de la parte más pequeña del software, como una función o método, para garantizar que se comporte según lo esperado.


31. Prueba de Integración

Proceso que verifica que los distintos módulos del sistema funcionen correctamente al combinarse.


32. Prueba de Aceptación

Validación final realizada por el cliente o usuario para comprobar que el software satisface sus necesidades.


33. Prueba de Rendimiento

Análisis que mide la capacidad del sistema para responder en términos de velocidad, estabilidad y consumo de recursos.


34. Depuración

Proceso de identificar, analizar y corregir errores o fallos en el código fuente del software.


35. Control de Versiones

Sistema que gestiona los cambios en el código fuente a lo largo del tiempo, permitiendo rastrear, revertir y colaborar en el desarrollo.


36. Git

Es una herramienta de control de versiones distribuido que permite a los desarrolladores colaborar, fusionar ramas y mantener historial de cambios.


37. Repositorio

Espacio central donde se almacena y gestiona el código fuente de un proyecto, ya sea local o en plataformas como GitHub o GitLab.


38. Rama (Branch)

Línea de desarrollo independiente dentro de un repositorio que permite trabajar en nuevas características sin afectar la rama principal.


39. Fusión (Merge)

Acción de combinar los cambios de una rama con otra dentro de un repositorio, integrando las modificaciones.


40. Integración Continua

Práctica de desarrollo que consiste en integrar cambios de código de manera frecuente y automática, acompañada de pruebas para detectar errores rápidamente.


41. Entrega Continua

Práctica de desarrollo que asegura que el software siempre esté en un estado desplegable, permitiendo liberaciones frecuentes y confiables.


42. Despliegue Continuo

Extiende la entrega continua al automatizar el paso de poner el software en producción después de pasar todas las pruebas.


43. DevOps

Conjunto de prácticas que integran desarrollo y operaciones para mejorar la colaboración, automatización y entrega de software.


44. Documentación de Software

Conjunto de manuales, diagramas y guías que explican el funcionamiento, diseño y uso de un sistema para facilitar su mantenimiento y comprensión.


45. Diseño de Interfaz de Usuario (UI)

Proceso de crear las pantallas, botones y elementos visuales con los que interactúa el usuario dentro de una aplicación.


46. Experiencia de Usuario (UX)

Percepción y satisfacción que obtiene el usuario al interactuar con un software, abarcando facilidad de uso, eficiencia y emociones.


47. Mantenimiento Correctivo

Tipo de mantenimiento que corrige errores detectados en el software después de su implementación.


48. Mantenimiento Perfectivo

Consiste en mejorar y optimizar el sistema agregando nuevas funciones o ajustando las ya existentes.


49. Mantenimiento Preventivo

Acciones realizadas para anticipar fallos y garantizar la estabilidad del sistema a largo plazo.


50. Mantenimiento Adaptativo

Cambios en el software para que continúe funcionando ante modificaciones en el entorno, como nuevos sistemas operativos o hardware.


51. Métricas de Software

Indicadores cuantitativos que permiten evaluar la calidad, rendimiento y productividad en el desarrollo de un sistema.


52. Calidad del Software

Grado en que un sistema cumple con los requisitos funcionales y no funcionales, asegurando confiabilidad y satisfacción del usuario.


53. Ingeniería de Software

Disciplina que aplica principios científicos, matemáticos y de ingeniería para el diseño, desarrollo, pruebas y mantenimiento de software.


54. Análisis de Factibilidad

Evaluación previa que determina si un proyecto de software es viable técnica, económica y operativamente.


55. Plan de Proyecto

Documento que organiza el cronograma, recursos, riesgos y actividades necesarias para desarrollar un sistema de software.


56. Diagrama de Actividades

Representación UML que describe el flujo de procesos o actividades dentro de un sistema, mostrando decisiones y paralelismos.


57. Diagrama de Secuencia

Modelo UML que muestra cómo los objetos interactúan entre sí a lo largo del tiempo mediante mensajes.


58. Diagrama de Componentes

Representación UML que ilustra los módulos de software y cómo se relacionan e integran entre sí.


59. Diagrama de Despliegue

Diagrama UML que muestra la disposición física de los nodos de hardware y la forma en que se ejecutan los componentes de software.


60. Especificación de Requisitos de Software (ERS)

Documento formal que detalla de manera estructurada los requisitos funcionales y no funcionales de un sistema.


61. Historia de Usuario

Descripción breve escrita desde la perspectiva del usuario que expresa una necesidad o funcionalidad requerida en el sistema.


62. Criterios de Aceptación

Condiciones específicas que deben cumplirse para que una historia de usuario o funcionalidad se considere terminada y validada.


63. Refinamiento del Backlog

Actividad periódica en Scrum donde se revisan, detallan y priorizan los elementos del Product Backlog.


64. Diseño Responsivo

Enfoque de diseño de interfaces que asegura que una aplicación se adapte correctamente a distintos tamaños de pantalla y dispositivos.


65. Arquitectura en Capas

Modelo de organización del software que separa la lógica en diferentes niveles, como presentación, negocio y datos, para mayor modularidad.


66. Arquitectura Cliente-Servidor

Estructura en la que un servidor central proporciona servicios y un cliente los solicita e interactúa con ellos.


67. API (Interfaz de Programación de Aplicaciones)

Conjunto de definiciones y protocolos que permiten que diferentes sistemas o aplicaciones se comuniquen entre sí.


68. Servicio Web

Mecanismo que permite a diferentes aplicaciones intercambiar datos a través de protocolos estándar como HTTP, SOAP o REST.


69. REST (Representational State Transfer)

Estilo arquitectónico que utiliza métodos HTTP (GET, POST, PUT, DELETE) para la comunicación entre cliente y servidor.


70. SOAP (Simple Object Access Protocol)

Protocolo basado en XML que define cómo intercambiar mensajes estructurados entre aplicaciones distribuidas.


71. Microservicios

Arquitectura de software donde las aplicaciones se dividen en pequeños servicios independientes que se comunican entre sí.


72. Contenedor de Software

Unidad ligera que empaqueta aplicaciones y sus dependencias, garantizando que se ejecuten igual en distintos entornos.


73. Docker

Plataforma que permite crear, desplegar y ejecutar aplicaciones en contenedores, facilitando la portabilidad y consistencia.


74. Kubernetes

Sistema de orquestación de contenedores que automatiza el despliegue, escalado y gestión de aplicaciones distribuidas.


75. Nube (Cloud Computing)

Modelo de servicios que ofrece recursos informáticos como almacenamiento, servidores o bases de datos a través de internet.


76. Plataforma como Servicio (PaaS)

Modelo en la nube que proporciona entornos de desarrollo completos para crear, probar y desplegar aplicaciones sin gestionar infraestructura.


77. Infraestructura como Servicio (IaaS)

Modelo en la nube que ofrece recursos de hardware virtualizados como servidores, redes y almacenamiento.


78. Software como Servicio (SaaS)

Modelo en el que el software se ofrece como un servicio accesible por internet, sin necesidad de instalarlo localmente.


79. Escalabilidad

Capacidad de un sistema para aumentar su rendimiento y manejar mayor carga de trabajo al añadir más recursos.


80. Alta Disponibilidad

Característica de un sistema diseñado para minimizar tiempos de inactividad y asegurar que esté accesible de manera constante.


81. Seguridad Informática

Conjunto de prácticas y tecnologías destinadas a proteger el software y los datos contra accesos no autorizados, ataques y pérdidas.


82. Cifrado de Datos

Proceso de transformar la información en un formato ilegible para quienes no poseen la clave de descifrado, garantizando confidencialidad.


83. Autenticación

Mecanismo que valida la identidad de un usuario o sistema antes de permitirle el acceso a recursos.


84. Autorización

Proceso que determina qué acciones o recursos puede acceder un usuario autenticado dentro de un sistema.


85. Prueba de Seguridad

Evaluación destinada a detectar vulnerabilidades en el software para prevenir riesgos de ataques o fugas de datos.


86. Arquitectura Orientada a Servicios (SOA)

Modelo en el que los componentes de software se exponen como servicios reutilizables que pueden interactuar entre sí.


87. Middleware

Software que actúa como intermediario entre aplicaciones, bases de datos o sistemas, facilitando la comunicación y la integración.


88. Base de Datos Relacional

Sistema de almacenamiento de información estructurada en tablas con filas y columnas, conectado mediante claves primarias y foráneas.


89. Base de Datos NoSQL

Modelo de almacenamiento no estructurado, diseñado para manejar grandes volúmenes de datos distribuidos y no tabulares.


90. Normalización de Bases de Datos

Proceso de organización de datos en tablas para reducir redundancias y mejorar la consistencia.


91. SQL (Structured Query Language)

Lenguaje estándar para la manipulación, consulta y gestión de bases de datos relacionales.


92. Big Data

Conjunto de datos masivos, variados y en constante crecimiento que requieren técnicas avanzadas de almacenamiento y análisis.


93. Inteligencia Artificial en el Desarrollo

Uso de algoritmos y modelos inteligentes para optimizar procesos de análisis, pruebas y mantenimiento en software.


94. Aprendizaje Automático (Machine Learning)

Subcampo de la inteligencia artificial que permite que los sistemas aprendan patrones y mejoren su desempeño con experiencia.


95. Ingeniería de Requisitos

Disciplina que se centra en la obtención, análisis, especificación y validación de los requisitos de un sistema.


96. Gestión de Configuración de Software

Práctica que asegura el control de versiones, cambios y estado del software durante todo su ciclo de vida.


97. Proceso Unificado (RUP)

Metodología de desarrollo iterativa que divide el ciclo de vida en fases: inicio, elaboración, construcción y transición.


98. Testing Automatizado

Uso de herramientas y scripts para ejecutar pruebas de software sin intervención manual, acelerando la validación.


99. Refactorización

Proceso de mejorar la estructura interna del código sin cambiar su comportamiento externo, para hacerlo más legible y mantenible.


100. Métricas de Calidad de Código

Conjunto de indicadores como complejidad ciclomática, duplicidad y cobertura de pruebas que permiten evaluar la salud del software.


101. Desarrollo Iterativo

Metodología en la que el software se construye en ciclos repetitivos, permitiendo ajustes basados en retroalimentación constante.


102. Desarrollo Incremental

Estrategia en la que el sistema se construye por partes, entregando incrementos funcionales en cada fase.


103. Prototipado Rápido

Técnica de crear versiones iniciales del software de forma veloz para validar requisitos y recoger opiniones de usuarios.


104. Validación de Requisitos

Proceso que asegura que los requisitos definidos realmente cumplen las necesidades del cliente y del negocio.


105. Verificación de Software

Conjunto de actividades que evalúan si el software cumple con lo que fue especificado en su diseño y documentación.


106. Benchmarking de Software

Evaluación comparativa del rendimiento de un sistema frente a estándares o competidores.


107. Escenarios de Prueba

Conjunto de situaciones diseñadas para evaluar el comportamiento del software ante distintos usos posibles.


108. Caso de Prueba

Documento específico que define entradas, condiciones y resultados esperados para comprobar una funcionalidad concreta.


109. Plan de Pruebas

Documento que establece el enfoque, alcance, recursos y calendario de las pruebas del software.


110. Trazabilidad de Requisitos

Relación que vincula cada requisito con su implementación, pruebas y validaciones correspondientes.


111. Gestión de Riesgos en Proyectos

Identificación, análisis y mitigación de riesgos que puedan afectar el éxito de un desarrollo de software.


112. Plan de Gestión de Riesgos

Documento que organiza estrategias de prevención y respuesta ante posibles problemas en un proyecto.


113. Estimación de Esfuerzo

Cálculo del tiempo y recursos necesarios para realizar las actividades de un proyecto de software.


114. Puntos de Función

Método de estimación que mide el tamaño del software en función de las funcionalidades que ofrece al usuario.


115. Estimación COCOMO

Modelo de estimación de costos de software que calcula esfuerzo, tiempo y personal según el tamaño del proyecto.


116. Cronograma de Proyecto

Planificación temporal de actividades y entregables, organizada en tareas con dependencias.


117. Diagrama de Gantt

Herramienta de gestión visual que representa el cronograma de actividades y su duración en el tiempo.


118. Hito de Proyecto

Evento clave que marca el cumplimiento de una fase o entregable importante en el desarrollo.


119. Gestión de Configuración

Práctica que asegura el control y la integridad de los artefactos de software a lo largo de su ciclo de vida.


120. Línea Base de Proyecto

Versión aprobada de un conjunto de documentos, código o requisitos que sirve como referencia oficial.


121. Estándares de Codificación

Conjunto de reglas y convenciones que guían la escritura de código para garantizar legibilidad y consistencia.


122. Complejidad Ciclomática

Métrica que mide la complejidad de un programa contando los caminos lógicos independientes en su código.


123. Revisiones de Código

Proceso colaborativo en el que los desarrolladores revisan mutuamente su código para detectar errores y mejorar calidad.


124. Integración de Sistemas

Proceso de combinar diferentes subsistemas en un sistema completo y funcional.


125. Middleware de Integración

Capa intermedia que conecta aplicaciones heterogéneas, facilitando el intercambio de datos.


126. Gestión de Requisitos Ágiles

Práctica de administrar requisitos de forma flexible y adaptativa durante el ciclo de desarrollo ágil.


127. Roadmap de Producto

Documento estratégico que muestra la visión, objetivos y evolución prevista de un software en el tiempo.


128. Arquitectura Hexagonal

Modelo que separa la lógica central del sistema de sus interfaces externas para facilitar mantenibilidad.


129. Arquitectura Limpia

Enfoque que organiza el código en capas concéntricas para aislar reglas de negocio de detalles técnicos.


130. Arquitectura Basada en Eventos

Modelo en el que los componentes se comunican principalmente a través de eventos asíncronos.


131. Testing de Regresión

Tipo de prueba que verifica que nuevas modificaciones no afecten funcionalidades ya existentes.


132. Testing Exploratorio

Prueba dinámica en la que el evaluador diseña y ejecuta casos en tiempo real, adaptándose a lo que descubre.


133. Testing de Caja Negra

Prueba que evalúa la funcionalidad del software sin conocer su código interno.


134. Testing de Caja Blanca

Prueba que examina la estructura interna del código y los flujos de ejecución.


135. Testing Alfa

Prueba realizada por desarrolladores internos en un entorno controlado antes de liberar el producto.


136. Testing Beta

Prueba realizada por usuarios reales en un entorno similar al de producción para validar la experiencia.


137. Release Candidate

Versión casi final del software que está lista para ser liberada si no se encuentran errores críticos.


138. Entorno de Desarrollo

Espacio donde los programadores crean y prueban nuevas funcionalidades antes de integrarlas.


139. Entorno de Pruebas

Ambiente controlado que simula condiciones de producción para evaluar el software antes de su liberación.


140. Entorno de Producción

Sistema real donde el software se encuentra operativo y disponible para los usuarios finales.


141. Sandbox

Entorno aislado donde se prueba software o código sin afectar el sistema principal.


142. Continuous Monitoring

Práctica de supervisión constante de un software en producción para detectar problemas en tiempo real.


143. Logging

Proceso de registrar eventos y actividades del software en archivos o sistemas para análisis posterior.


144. Auditoría de Software

Proceso formal de revisión para asegurar que el software cumple con normas y estándares establecidos.


145. Métrica de Mantenibilidad

Indicador que mide la facilidad con que un software puede ser modificado o mejorado.


146. Prueba de Usabilidad

Evaluación de qué tan fácil resulta para los usuarios interactuar con el sistema.


147. Accesibilidad en Software

Capacidad de una aplicación para ser utilizada por personas con discapacidades físicas o cognitivas.


148. Internacionalización de Software

Proceso de preparar un sistema para soportar múltiples idiomas y formatos culturales.


149. Localización de Software

Adaptación de un sistema a un idioma, cultura o mercado específico.


150. Modularidad

Principio de diseño que divide el sistema en partes independientes que pueden desarrollarse y mantenerse separadamente.


151. Reutilización de Software

Uso de componentes existentes en nuevos proyectos para ahorrar tiempo y reducir costos.


152. Ingeniería Inversa

Proceso de analizar un software existente para descubrir su diseño, estructura y funcionamiento.


153. Reingeniería de Software

Proceso de modificar y mejorar un sistema existente sin alterar su funcionalidad principal.


154. Obsolescencia Tecnológica

Situación en la que un software deja de ser útil por falta de soporte o incompatibilidad con nuevas tecnologías.


155. Deuda Técnica

Consecuencia de tomar atajos en el desarrollo que generan dificultades futuras en mantenimiento y escalabilidad.


156. Métrica de Productividad

Medida que relaciona el esfuerzo invertido con la cantidad de funcionalidades desarrolladas.


157. Pair Programming

Práctica ágil en la que dos programadores trabajan juntos en una misma estación de trabajo: uno escribe código y el otro revisa.


158. Test Driven Development (TDD)

Práctica en la que se escriben pruebas antes de implementar el código para asegurar que cumpla con los requisitos.


159. Behavior Driven Development (BDD)

Extensión de TDD que fomenta la escritura de pruebas en lenguaje natural, facilitando la comunicación con el cliente.


160. Domain Driven Design (DDD)

Enfoque que centra el diseño del software en el modelo del dominio del negocio para mayor alineación con la realidad.


161. Continuous Delivery Pipeline

Conjunto de etapas automatizadas para construir, probar y desplegar software de forma continua.


162. Feature Toggle

Técnica que permite activar o desactivar funcionalidades en producción sin modificar el código.


163. Smoke Testing

Prueba inicial rápida que evalúa si el sistema arranca y funciona de manera básica.


164. Pruebas de Carga

Evaluación de cómo responde el sistema cuando se somete a grandes volúmenes de usuarios o transacciones.


165. Pruebas de Estrés

Análisis que mide cómo se comporta el software bajo condiciones extremas hasta llegar a fallar.


166. Pruebas de Escalabilidad

Examen que determina si el software mantiene su rendimiento al aumentar progresivamente la carga.


167. API Testing

Proceso de evaluar las interfaces de programación para garantizar que devuelvan respuestas correctas y seguras.


168. Integración Continua con Jenkins

Uso de la herramienta Jenkins para automatizar la compilación, pruebas e integración del software.


169. Control de Incidencias

Proceso de registrar, monitorear y resolver errores o problemas detectados durante el ciclo de vida del software.


170. Help Desk

Servicio de soporte al usuario que atiende consultas y reportes relacionados con el funcionamiento del sistema.


171. SLA (Service Level Agreement)

Acuerdo formal que define los niveles de servicio que un proveedor debe garantizar a sus clientes.


172. KPI en Proyectos de Software

Indicadores clave de rendimiento que miden el éxito del desarrollo en tiempo, calidad y costos.


173. Métricas de Código Limpio

Indicadores que evalúan legibilidad, consistencia y simplicidad del código escrito.


174. Refactoring de Arquitectura

Proceso de reorganizar la estructura general del sistema sin alterar su comportamiento funcional.


175. Prueba de Compatibilidad

Verificación de que el software funciona en diferentes dispositivos, sistemas operativos y navegadores.


176. Testing Multiplataforma

Estrategia que evalúa si una aplicación mantiene la misma funcionalidad en diversos entornos.


177. Virtualización de Entornos

Uso de máquinas virtuales para simular entornos de ejecución y pruebas de software.


178. Emulación de Dispositivos

Simulación de dispositivos físicos para probar software en condiciones cercanas a la realidad.


179. Testing en la Nube

Práctica que utiliza servicios cloud para ejecutar pruebas de manera escalable y flexible.


180. Monitoreo de Aplicaciones (APM)

Uso de herramientas que miden rendimiento, disponibilidad y experiencia del usuario en tiempo real.


181. Pipeline como Código

Definición de flujos de integración y entrega continua mediante archivos de configuración versionados.


182. Versionado Semántico

Convención que utiliza números de versión para indicar cambios mayores, menores y parches en el software.


183. Release Notes

Documento que acompaña una nueva versión de software y detalla sus cambios, mejoras y correcciones.


184. Hotfix

Corrección urgente que se aplica directamente en producción para resolver un error crítico.


185. Patch

Actualización que corrige errores menores o vulnerabilidades en un software existente.


186. Service Pack

Conjunto acumulado de actualizaciones y parches lanzados para un software.


187. Rolling Deployment
